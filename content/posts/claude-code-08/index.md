# Agent 如何记住"上次说过的话"？

单次对话结束后，Claude Code 默认什么都不记得。但工程实践中，很多知识需要跨越会话持久存在——项目约定、架构决策、踩过的坑。这一章梳理 Claude Code 的四层记忆架构，以及如何用好它、突破它的局限。

## 8.1 四层记忆架构总览

Claude Code 的记忆系统由四层构成，每一层的持久性、作用范围和更新方式各不相同。理解这四层，是有效使用 Claude Code 的基础。

| 层级       | 载体                  | 持久性                 | 作用范围           | 更新方式          |
| ---------- | --------------------- | ---------------------- | ------------------ | ----------------- |
| 企业全局层 | `~/.claude/CLAUDE.md` | 永久                   | 当前用户所有项目   | 手动编辑          |
| 项目层     | `./CLAUDE.md`         | 永久（git 追踪）       | 当前 repo 所有会话 | 手动 / Agent 写入 |
| 子目录层   | `./src/CLAUDE.md`     | 永久（git 追踪）       | 该目录下的操作     | 手动 / Agent 写入 |
| 会话层     | 对话历史（内存）      | 临时（会话结束即清除） | 当前对话           | 自动累积          |

### 优先级规则

内层覆盖外层。当企业全局层说"使用 2 空格缩进"，但项目层说"使用 4 空格缩进"，以项目层为准。这与 Unix 配置文件的经典设计（`/etc` → `~/.config` → 本地）完全一致，工程师对这个心智模型已经非常熟悉。

### 加载时机

Claude Code 在每次会话开始时，按照从外到内的顺序加载所有 CLAUDE.md 文件，将它们注入到 System prompt 之后、对话历史之前。这意味着 CLAUDE.md 的内容是"永远存在"的——每次对话它都在那里，不会被压缩掉。

## 8.2 CLAUDE.md 深度设计

模块三已经介绍过 CLAUDE.md 的基本用法。这里深入讲两个在实践中容易忽视的设计细节：**自动维护** 和 **信息衰减**。

### Agent 可以自己更新 CLAUDE.md

CLAUDE.md 不必完全由人工维护。Agent 在完成探索任务后，可以被指示把发现写入 CLAUDE.md：

```
完成后，把这次发现的架构约定和踩到的坑写进 CLAUDE.md，
以便下次对话直接可用。
```

这形成了一个"自我进化的知识库"——每次 Agent 完成一个任务，都在为未来的自己留下更好的起点。

### 信息衰减问题

CLAUDE.md 有一个容易忽视的问题：**写进去的内容不会自动过时**。代码在演进，但文档可能停留在三个月前的状态。过时的 CLAUDE.md 比没有 CLAUDE.md 更危险——它会给 Agent 错误的"基础事实"。

管理信息衰减的几种策略：

**时间戳标注**

在 CLAUDE.md 的关键条目旁注明写入时间，定期审查超过特定时间的条目是否仍然有效。

```markdown
## 架构约定
- 认证走 src/auth/（2025-01）
- DB 连接池单例 src/db/pool.ts（2025-01）⚠️ 待验证
```

**主动失效声明**

对于可能变化的内容，明确写出"该信息在什么条件下失效"：

```markdown
## 已知问题
- token.test.ts 有 2 个用例失败（bug #342，预计 v2.1 修复后可删除此条）
```

**定期 Agent 审查**

定期让 Agent 扫描 CLAUDE.md 并对照当前代码库，标记可能过时的条目：

```
请检查 CLAUDE.md 里的所有条目，对照当前代码库，
标出哪些可能已经过时，哪些需要更新。
```

## 8.3 会话记忆：对话历史的生命周期

会话层记忆是最动态、也最脆弱的一层。它在对话开始时为空，随着每轮 Thought-Action-Observation 循环不断增长，会话结束后消失。

### 对话历史的结构

每一轮交互都以固定格式追加到历史中：

```
[用户输入]
[Assistant Thought]
[Tool Call: name + params]
[Tool Result: stdout/stderr]
[Assistant Thought]
...
[Assistant 最终回复]
```

这个结构是线性的、不可修改的——你无法"删除"某一轮对话，也无法"插入"过去某个时间点的信息。历史只会增长，不会收缩（除非触发压缩）。

### Session 持久化

会话可以跨进程持久化。Claude Code 将对话历史序列化到磁盘，通过以下方式恢复：

```bash
# 自动接续上次未完成的会话
claude --continue

# 列出历史会话，选择恢复
claude --resume

# 在新任务中引用上次会话
claude --resume <session-id>
```

**持久化的局限**：恢复的历史是"冷"的。模型重新加载这段历史时，对其中细节的"激活程度"不如刚刚执行时高——类似于人类重新阅读一段旧笔记，理解它需要更多认知资源。对于非常长的任务，持续写入 CLAUDE.md 比单纯依赖历史持久化更可靠。

## 8.4 压缩与信息损失

当对话历史增长到接近上下文窗口上限时，Claude Code 会触发压缩（compaction）。这是记忆系统里最需要理解的机制，也是最容易产生问题的地方。

### 压缩做了什么

压缩的本质是"有损摘要"：把历史中的详细内容替换为一段摘要文本，保留任务目标和关键结论，丢弃中间过程的细节。

```
原始历史（4,000 tokens）：
  Turn 1: 读取 package.json → 发现用 vitest
  Turn 2: 运行测试 → 17 个通过，2 个失败
  Turn 3: 读取 token.ts → 发现第 24 行的 bug
  Turn 4: 修改 token.ts → diff 展示
  Turn 5: 再次运行测试 → 全部通过
  ...（共 40 轮）

压缩后（800 tokens）：
  [摘要] 已完成：修复 token.ts 的 JWT 验证 bug（第 24 行），
  测试全部通过。当前状态：所有 auth 模块测试绿色。
  待完成：用户要求的第二个任务（添加刷新 token 逻辑）。
```

### 什么信息容易丢失

**中间过程的探索结果**：Agent 在第 3 轮发现"数据库连接池是单例"这个细节，在第 20 轮可能需要用到，但压缩后这个细节可能消失，Agent 不得不重新发现它。

**负面知识**：Agent 试过某个方案但失败了——这个"踩坑"记录在历史里，压缩后可能变成空白，Agent 可能重蹈覆辙。

**精确的数字和标识符**：具体的端口号、UUID、版本号这类精确信息，在摘要中容易被泛化或丢失。

### 减轻信息损失的实践策略

**在压缩前写入 CLAUDE.md**

每完成一个阶段性任务后，主动把关键发现写进 CLAUDE.md——这相当于把重要信息从"会话记忆"升级到"持久记忆"，不受压缩影响。

```
在继续下一个任务之前，把这次发现的所有关键信息
（命令、约定、已知问题）更新到 CLAUDE.md。
```

**使用 `/compact` 手动触发**

在任务的阶段性完成点手动触发 `/compact`，可以让压缩发生在"信息完整"的状态，而不是等到快溢出时被动压缩。主动压缩的摘要质量通常优于被动压缩。

**显式记录关键状态**

对于重要的中间结果，让 Agent 写成结构化的临时文件（`/tmp/task-state.md`）。这些文件不受上下文压缩影响，需要时可以重新读入。

```bash
# Agent 在执行中创建状态文件
Write /tmp/session-state.md:
  当前进度：已完成认证模块，待完成：支付模块
  已知坑：stripe-sdk v4 与 node 20 有兼容问题，用 v3.x
  关键文件：src/payment/webhook.ts（核心逻辑在此）
```

## 8.5 长期记忆的局限与外部知识库

CLAUDE.md 解决了跨会话的"项目知识"存储问题，但它有一个结构性局限：**它是静态文件，不能被检索，只能被全量加载**。

当 CLAUDE.md 变得很长（超过几千 token），它会消耗大量上下文空间，而其中大部分内容在具体任务中可能并不相关。理想的长期记忆应该是"按需检索"而非"全量加载"——这就是外部知识库（RAG）能解决的问题。

### RAG 接入的思路

目前 Claude Code 原生不支持向量数据库检索，但可以通过 MCP server 实现：

**方案一：MCP + 向量数据库**

构建一个 MCP server，背后连接 Pinecone、Chroma 或 pgvector，提供语义搜索工具。Agent 在需要查找历史知识时，调用这个工具检索相关内容，而不是把所有知识全部加载进上下文。

```typescript
// MCP tool 声明示例
{
  name: "search_project_knowledge",
  description: "语义搜索项目历史决策、架构约定和已知问题",
  inputSchema: {
    type: "object",
    properties: {
      query: { type: "string", description: "搜索内容" },
      top_k: { type: "number", default: 5 }
    }
  }
}
```

**方案二：结构化 CLAUDE.md + 分文件管理**

把 CLAUDE.md 按主题拆分到子目录：

```
docs/
├── CLAUDE.md              # 入口，只含运行命令和目录索引
├── claude/
│   ├── architecture.md    # 架构决策记录（ADR）
│   ├── known-issues.md    # 已知问题和临时 workaround
│   └── api-contracts.md   # 接口约定
```

在主 CLAUDE.md 里只放"元信息"和"指针"，Agent 按需读取具体文件。这不是真正的按需检索，但能显著减少默认加载的上下文体积。

### 记忆系统的演进方向

Claude Code 当前的记忆系统是务实的——用文件解决持久化，用上下文窗口解决会话内记忆，简单可靠。但随着任务复杂度增加，几个方向的演进是可以预见的：

**自动知识提取**：任务完成后自动识别值得持久化的知识，不需要用户手动指示 Agent 更新 CLAUDE.md。

**增量式代码库索引**：在后台持续维护代码库的语义索引，Agent 需要查找信息时直接检索，而不是实时执行 Glob/Grep。

**记忆的主动失效**：基于代码变更自动标记相关 CLAUDE.md 条目为"待验证"，减少信息衰减问题。

> **记忆系统设计的核心矛盾**：你希望 Agent 记住所有有用的东西，但上下文窗口是有限的。解决这个矛盾的方向不是扩大窗口（虽然这也有帮助），而是让记忆变得更"聪明"——知道什么时候该记什么，什么时候该忘什么，什么时候该去检索什么。

---

## 小结

模块八的核心是"知识的生命周期管理"：

- **四层架构** 解决的是"哪类知识放哪里"
- **CLAUDE.md 维护** 解决的是"如何让持久记忆不腐烂"
- **会话历史管理** 解决的是"单次任务内如何保持上下文连贯"
- **压缩策略** 解决的是"上下文溢出时如何最小化信息损失"
- **外部知识库** 解决的是"当项目知识量超过静态文件能承载的规模时怎么办"
