## 3.1 CLAUDE.md：Agent 的"项目记忆"

### 四层架构，从外到内

CLAUDE.md 采用嵌套覆盖的设计——越内层优先级越高，可以覆盖外层的配置：

```
┌─────────────────────────────────────────────────────┐
│  企业 / 全局  ~/.claude/CLAUDE.md · 所有项目共享      │
│  ┌───────────────────────────────────────────────┐   │
│  │  项目根目录  ./CLAUDE.md · 整个 repo 可见       │   │
│  │  ┌─────────────────────────────────────────┐  │   │
│  │  │  子目录  ./src/CLAUDE.md · 局部覆盖      │  │   │
│  │  │  ┌───────────────────────────────────┐  │  │   │
│  │  │  │  会话级对话历史（运行时动态）        │  │  │   │
│  │  │  └───────────────────────────────────┘  │  │   │
│  │  └─────────────────────────────────────────┘  │   │
│  └───────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
              优先级：内层 > 外层
```

| 层级         | 路径                  | 适合写的内容                             |
| ------------ | --------------------- | ---------------------------------------- |
| 企业全局层   | `~/.claude/CLAUDE.md` | 公司通用规范、禁止提交 API 密钥的提醒    |
| 项目根目录层 | `./CLAUDE.md`         | 构建命令、架构说明、已知技术债           |
| 子目录层     | `./src/CLAUDE.md`     | 该模块特有的约定（如"纯函数，无副作用"） |
| 会话动态层   | 对话历史本身          | 本次任务的临时约束                       |

### CLAUDE.md 示例

```markdown
# 我的项目名 — CLAUDE.md

## 运行命令
- 开发服务器：`npm run dev`（端口 3000）
- 单测：`npm test`，集成测试：`npm run test:e2e`
- 构建：`npm run build`，类型检查：`npm run typecheck`

## 架构约定
- API 路由在 `src/routes/`，每个文件对应一个资源
- 数据库通过 `src/db/pool.ts` 的单例访问，禁止直连
- 所有错误统一通过 `AppError` 类抛出

## 注意 / 禁区
- `src/legacy/` 目录不要碰，老代码，碰了会出问题
- `token.test.ts` 里有 2 个测试本来就失败（已知 bug #342）
- 提交前必须跑 `npm run lint`
```

### 该写什么 vs. 不该写什么

**该写**：可直接执行的命令、不明显的架构约定、禁区和已知地雷、重要的技术决策背景、团队特有的缩写或术语。

**不该写**：能从代码直接读出的信息、大篇幅的"项目介绍"叙事、过时的内容、无关 Agent 操作的背景故事、冗长的注释和解释。

> **CLAUDE.md 的核心价值：把"隐性知识"变成"显性配置"。那些只有老工程师才知道的项目约定、坑和禁区，通过 CLAUDE.md 变成 Agent 的第一手知识。**

## 3.2 代码库探索策略：按需拉取

### 六步探索流程

Agent 接手陌生项目的标准流程：

**第一步：鸟瞰目录结构**

```bash
# 只看两层，避免噪音
LS . --depth=2
```

目录结构是"代码库的骨架"，30 秒扫一眼就能判断项目类型，是后续所有决策的基础。

**第二步：读入口文档**

```bash
Read README.md
Read CLAUDE.md   # 如果存在
Read package.json
```

`package.json` 的 `scripts` 字段是"项目操作手册"，一读就知道如何跑测试、如何构建。

**第三步：定位入口文件**

```bash
# 从 package.json 得知入口是 src/index.ts
Read src/index.ts
```

入口文件就像小说的第一章——它告诉你整个系统由哪些"章节"构成，以及它们的组织方式。

**第四步：按需 Grep 关键符号**

```bash
Grep "export.*setupRoutes" src/ --include="*.ts"
Grep "initDB\|createPool" src/db/ -n
```

Grep 是"精准手术刀"。与其读完整个 routes 目录，不如直接搜到定义的位置，按需取用。

**第五步：运行项目验证理解**

```bash
Bash: npm test -- --run 2>&1 | tail -30
# 或: python -m pytest tests/ -x -q
```

失败的测试是"已知地雷图"。Agent 在执行任何修改之前先了解哪些测试本来就失败，才能判断自己的改动有没有引入新问题。

**第六步：写入 CLAUDE.md 建立记忆**

```markdown
## 运行命令
- 开发: `npm run dev`
- 测试: `npm test`

## 架构约定
- 路由在 src/routes/
- DB 连接池单例：src/db/pool.ts

## 注意
- token.test.ts 的 2 个用例本来就失败（已知 bug）
```

一旦写好，Agent 在这个项目里的每次对话都能直接使用这些知识，而不必每次重新探索。

### Pull，而不是 Push

Agent 永远是"拉取"需要的信息，而不是"推送"所有信息进来：

- **推送模式（避免）**：先把 src/ 下所有 .ts 文件全部读入上下文，然后开始思考任务。100 个文件 = 立刻消耗大量 token，且 90% 是无用的。
- **拉取模式（使用）**：只读入口文件，识别关键模块名，再用 Grep 定位具体符号，最后只读那一个函数的实现。10 个文件 = 满足需求。

> **经验法则：能用 Grep 找到的，就不要用 Read 读。Read 是"读全文"，Grep 是"找答案"。**

## 3.3 大型 Monorepo 的挑战

| 挑战         | 描述                                               |
| ------------ | -------------------------------------------------- |
| 目录结构太深 | LS 两层只能看到顶层，需要多次探索才能建立全局图景  |
| 符号名称冲突 | 多个包都有 `utils.ts`，Grep 可能返回几十个匹配     |
| 跨包依赖追踪 | 跨包的调用链追踪远比单一 repo 复杂，容易丢失上下文 |
| 测试矩阵爆炸 | 全量测试耗时长，需精准指定 `--filter` 只跑相关包   |

### 应对策略

- 在 CLAUDE.md 里明确标注各子包的职责和入口
- 用子目录级 CLAUDE.md 为每个子包单独写入约定
- 任务开始时明确告诉 Agent 工作范围："只改 `packages/api` 下面的内容"

> **注意**：如果项目超过约 500 个有效源文件，建议人工辅助缩小任务范围，不要让 Agent 自己"摸索全局"——代价太高，效果也不稳定。

## 3.4 多语言与多框架适配

### 语言识别：靠文件信号

Agent 识别语言和框架完全基于文件名和内容的"信号"：

| 语言 / 框架 | 主要信号文件                                       | 推断的命令                                  |
| ----------- | -------------------------------------------------- | ------------------------------------------- |
| Python      | `pyproject.toml` / `setup.py` / `requirements.txt` | `pytest`、`python -m`、`pip install`        |
| TypeScript  | `package.json` + `tsconfig.json`                   | 读 scripts 字段；`npm` / `pnpm` / `yarn`    |
| Go          | `go.mod`                                           | `go build`、`go test ./...`、`go run .`     |
| Rust        | `Cargo.toml`                                       | `cargo build`、`cargo test`、`cargo clippy` |
| Ruby        | `Gemfile` + `Rakefile`                             | `bundle exec rspec`、`rails s`、`rake`      |

### 框架适配：理解约定而非代码

Claude Code 的多框架适配，本质上是模型训练时对各框架文档和代码的大量学习——它"见过"太多 Next.js、Django、Spring Boot 项目，能从文件结构和命名约定快速推断框架，然后应用相应知识。

> **微妙之处**：模型对"流行框架"的理解远深于"冷门框架"。如果你在用小众框架或公司内部定制脚手架，这时 CLAUDE.md 里写清楚"这个项目用的是内部框架 X，路由注册方式是 Y"，就尤为重要。

### 语言无关的抽象层

Claude Code 的精妙之处在于，核心工具（`Bash`、`Read`、`Write`、`Grep`）本身完全语言无关。Agent 不需要"知道怎么操作 Python 文件"——它只需要知道"运行 `pytest` 这个命令"、"读这个 .py 文件"。

这就是 Terminal-first 设计的另一个隐藏红利：一套工具集，支持世界上所有语言。只要这门语言有命令行工具链，Claude Code 就能驾驭它——哪怕是 COBOL、Fortran，或者明天发明的新语言。
