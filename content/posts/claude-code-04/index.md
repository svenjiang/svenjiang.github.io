## 4.1 任务分解：拆开问题的艺术

用户给出的任务往往是一个"目标"，而不是一个"步骤列表"。"帮我给项目加 CI"是目标；创建目录、写 YAML、验证语法、检查 Node 版本才是步骤。Agent 需要自己完成这个转换。

### 什么时候值得分解

任务分解本身有成本——Agent 需要花推理资源来规划，而不是直接行动。所以不是所有任务都值得分解。Claude Code 做了一个隐式的复杂度判断：

| 任务类型             | 处理方式     | 示例                                         |
| -------------------- | ------------ | -------------------------------------------- |
| 单步骤，目标清晰     | 直接执行     | 给函数加注释、改变量名                       |
| 多组件，步骤间有依赖 | 先分解再执行 | 加 CI 配置、实现完整功能模块、跨文件重构接口 |

### 分解的三个维度

**按组件拆分**

把一个大功能拆成独立的子模块。"加 CI"→ 测试 workflow + lint workflow + 验证，三个独立组件互不干扰，可以并行。

**按依赖排序**

有些步骤必须先做，有些可以并行。先探索项目结构、确认工具链，再动手写文件——顺序不能颠倒。

**加入验证步骤**

每个子任务完成后，Agent 会主动加入一个"验证"动作——运行测试、检查语法、读一下写入的文件是否符合预期。这是"自我质检"。

> **任务分解是隐式的，不是显式的。** Agent 不会真的打印出一份"执行计划"给用户看（除非你要求它这么做）。规划发生在 Thought 阶段，用户看到的只是一系列流畅的工具调用。这与人类工程师的工作方式类似——在脑子里过一遍计划，然后开始动手。

## 4.2 执行中的决策：何时问，何时做，何时重试

执行过程中不可避免地会遇到两类问题：歧义（不知道该怎么做）和错误（做了但没成功）。这两类问题需要截然不同的应对策略，也是 Agent 行为质量差异最大的地方。

### 歧义 → 询问用户

歧义是"需求不明确"，不是执行失败。Agent 面对歧义的正确反应是停下来问，而不是猜一个"看起来合理"的答案。

> **问用户的时机判断标准**：如果不同选择会导致"用户资源消耗""行为不可逆"或"影响范围超出当前文件"的差异，就应该问。如果只是"两种写法都可以"的风格差异，自行选一个更合理的即可。

### 执行中的三种情况对照

| 遇到的情况                 | Agent 的处理 | 依据                                    |
| -------------------------- | ------------ | --------------------------------------- |
| 需求有多种理解，影响结果   | ⚠️ 询问用户   | Minimal Footprint：关键决策交还用户     |
| 语法错误、工具报错         | 🔄 自行修复   | 有明确的正确答案，无需用户介入          |
| 文件已存在，是否覆盖       | ⚠️ 询问用户   | 不可逆操作，必须确认                    |
| 风格选择（tabs vs spaces） | ✅ 自行决定   | 读 `.editorconfig` / 项目约定后自行对齐 |
| 依赖版本冲突               | ⚠️ 询问用户   | 影响整个项目依赖树，用户需知晓          |
| 测试失败（新引入的）       | 🔄 自行修复   | 这是 Agent 应该解决的问题               |

### 错误恢复的层次

Agent 遭遇执行错误时，会按照"代价递增"的顺序尝试恢复：

1. **原地重试** — 偶发性失败（网络超时、文件锁），再试一次就好。
2. **调整参数** — 读报错信息，修改命令参数或文件内容，重新执行。
3. **换路径** — 当前方法不可行时，换一种实现方式。
4. **告知用户** — 以上都失败，或恢复需要用户授权时，停下来汇报现状。

> **注意**：Agent 不会无限重试。连续失败超过一定次数后，它会停止并向用户说明"我尝试了 X 种方式都没有成功，需要你来看一下"。这个"知道自己卡住了"的元认知能力，比盲目重试更重要。

## 4.3 并行 Agent：Task 工具

### Task 工具做了什么

调用 Task 工具时，主 Agent 会启动一个独立的子 Agent 实例，把子任务的描述传给它，然后继续执行其他任务。子 Agent 拥有自己独立的上下文窗口，在自己的 ReAct 循环里运行，完成后把结果返回给主 Agent。

| 角色                     | 职责                                                         |
| ------------------------ | ------------------------------------------------------------ |
| 主 Agent（Orchestrator） | 整体规划、分发子任务、等待结果、汇总输出，不直接执行文件读写 |
| 子 Agent（Subagent）     | 接收具体子任务描述，独立执行完整的 ReAct 循环，返回最终结果  |

### 并行的前提：任务独立性

Task 并行有一个硬性前提——子任务之间不能有依赖关系。"子任务 A 的输出是子任务 B 的输入"这种情况，不能并行，必须串行。Agent 在分解任务时需要先判断依赖关系，再决定哪些可以并行。

> **并行的真正价值**体现在 IO 密集型任务上，比如同时搜索多个代码库目录、同时调用多个 WebFetch 抓取文档。对于 CPU 密集型（模型推理）任务，并行的收益受制于 API 并发限制。

### 共享状态的处理

子 Agent 之间没有共享内存——它们通过返回结果来传递信息，而不是直接读写同一个变量。这个设计简单但有效：避免了并发写同一个文件的冲突，也让每个子任务的边界更清晰。主 Agent 负责把各子任务的结果合并成最终输出。

## 4.4 "死胡同"检测

Agent 在长任务中有一类特殊的失败模式：它知道自己在执行某件事，但实际上已经陷入了无效循环——反复尝试同样的操作，每次都失败，但没有意识到这个模式本身就是问题所在。

### 典型的死胡同场景

**修测试场景**：Agent 改了代码使测试通过，但改动引入了新的测试失败，然后修新的失败又破坏了旧的……在没有全局视角的情况下，Agent 可能在这个局部循环里打转，无法收敛。

**依赖安装场景**：某个包安装失败，Agent 换了版本号，还是失败，再换……每次都是不同的版本，但根本原因（比如 Node 版本不兼容）始终没有被识别。

### 如何跳出

**识别重复模式**

Agent 在每次 Thought 阶段会检查：过去 N 轮里，我是否在做相同的事情？如果是，这个方向可能本身就行不通。

**回到更高层视角**

从局部问题退出来，重新审视整体目标。"我真正要解决的是什么？现在的做法和目标是否还对齐？"

**主动汇报并停止**

如果自行跳出无望，Agent 会停下来告诉用户："我在这里遇到了问题，尝试了 X 种方式都没有解决，需要你的判断。"这比继续无效循环好得多。

> **死胡同检测的本质是"元认知"**——Agent 不仅在执行任务，还在监控自己执行任务的过程是否有效。这个能力的成熟度，很大程度上决定了一个 Agent 在复杂任务上的可靠性上限。

## 小结

模块四的核心是"有规划的行动"：

- **任务分解** 解决的是"从目标到步骤"的转换
- **执行中的决策** 解决的是"遇到不确定性时怎么办"
- **并行 Agent** 解决的是"如何提升执行效率"
- **死胡同检测** 解决的是"如何不在无效路径上浪费时间"

四者合力，让 Agent 的执行从"线性跑命令"升级为"有策略的工程师行为"。
