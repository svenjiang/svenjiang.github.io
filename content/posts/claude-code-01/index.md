> 从 Copilot 式补全到真正的 Agent——一次关于"让 AI 真正干活"的深度思考。

## 1.1 为什么需要 Coding Agent？

### 补全模式的天花板

2021 年 GitHub Copilot 发布时，大多数人觉得 AI 辅助编程的终极形态已经到来：在 IDE 里边写边补，感觉像拥有了一个随时待命的结对编程伙伴。但几年用下来，一个根本性的局限逐渐浮现：

> **补全模式的本质局限：它只能帮你写下一行，却无法帮你"完成一件事"。**

### 补全 vs. 推理：两种不同的智能

如果你让 Copilot "把这个函数的参数校验补全"，它很擅长。但如果你说"我需要给这个项目加一套用户权限系统"——它就失效了。原因不是模型不聪明，而是**任务范式**完全不同：

| 任务类型           | 补全模式   | Agent 模式 |
| ------------------ | ---------- | ---------- |
| 补全一个函数体     | ✅ 擅长     | 大材小用   |
| 跨文件重命名变量   | ❌ 无法做到 | ✅ 擅长     |
| 修复一个测试失败   | ❌ 无法做到 | ✅ 擅长     |
| 理解整个代码库结构 | ❌ 无上下文 | ✅ 主动探索 |
| 运行命令验证结果   | ❌ 无执行权 | ✅ 擅长     |

补全模式的 AI 是一支非常聪明的"钢笔"——它能写出漂亮的句子，但它不知道你在写什么文章，也不会帮你查资料、改错别字、或者把第三章和第一章的逻辑对齐。

### 软件工程的本质是"迭代验证"

真正的编程工作流是：写代码 → 运行测试 → 看报错 → 定位问题 → 修改 → 再运行。这个循环一天可能要重复几十次。补全工具只参与了"写代码"这一步，而 Agent 能参与整个循环。

> **软件工程中大约 80% 的时间不是在"写新代码"，而是在理解现有代码、调试、重构和验证。Coding Agent 的真正价值，正是在这 80% 里。**

## 1.2 Claude Code 的设计定位

### 取舍一：Terminal-first，而非 IDE-first

大多数人的第一直觉是"把 Agent 嵌进 IDE"——Cursor 走的就是这条路。Claude Code 的选择截然相反：它是一个命令行工具。

为什么？不是因为 CLI 更酷，而是因为 **shell 才是操作系统的真正接口**。在终端里，Agent 可以：

- **运行任意命令**：git、npm、python、docker……不依赖 IDE 插件体系，任何已安装的工具都可直接调用。
- **访问完整文件系统**：不受 IDE 工作区限制，可以跨 repo、读取配置文件、访问环境变量。
- **无缝接入 CI/CD**：GitHub Actions、GitLab CI 等流水线本就是 shell 环境，CLI 工具可以零改动集成。

本质上，Terminal-first 是"工具普适性"的选择：牺牲了精美的 GUI，换来了在任何环境里都能工作的能力。

### 取舍二：真实权限，而非沙箱隔离

这是 Claude Code 最有争议的设计决定之一。它默认在真实系统上运行，能读写真实文件，能执行真实命令。

> 这不是"安全漏洞"，而是一个深思熟虑的权衡：一个被过度限制的 Agent，在遇到任何超出其沙箱的操作时都会卡住，反而没有实用价值。

### 取舍三：Minimal Footprint 原则

既然给了真实权限，Claude Code 要求自己遵守一套行为原则——尽量减少不必要的"足迹"（footprint）：

| 维度     | 原则                                       |
| -------- | ------------------------------------------ |
| 操作倾向 | 优先可逆操作，能用 git 管理的变更就用 git  |
| 权限申请 | 最小必要权限，不预先申请"以防万一"的权限   |
| 关键节点 | 在不可逆操作（删除、部署）前停下来询问用户 |
| 不确定时 | 宁可少做，多问——歧义任务不自行猜测         |

> **Minimal Footprint 的核心思想：Agent 的能力来自用户的授权，不是 Agent 自己的权利。任何时候用户都应该是真正的决策者。**

## 1.3 与同类产品的设计对比

Coding Agent 领域目前有几条主要路线。理解它们的差异，需要一个核心维度：**自主性（autonomy）与控制权（control）的平衡**。

```
← 用户控制                                          Agent 自主 →
    Copilot        Cursor      Claude Code    Aider       Devin
```

### Copilot / Copilot Chat：补全增强路线

本质是"更聪明的自动完成"，深度嵌入 IDE，用户始终主导每一次操作。优点是安全、可预期；局限是无法完成需要多步骤推理和执行的任务。

### Cursor：IDE 深度集成路线

在 VS Code 的基础上重构，提供了 Composer（多文件编辑）和 Agent 模式。兼顾了 GUI 体验和一定程度的自主性，是目前 UI 完成度最高的方案。代价是强绑定特定 IDE，CI/CD 场景支持较弱。

### Claude Code：Terminal + 中等自主路线

定位在"能独立完成一个功能分支"的自主程度，但始终保留清晰的人类确认节点。Terminal-first 使其在服务器、CI、headless 环境中如鱼得水。

### Aider：开源 CLI 路线

与 Claude Code 定位接近，同样是命令行工具，更偏向开发者自行配置。开源生态是其优势，但开箱体验和安全机制相对原始。

### Devin：全自主路线

Cognition 推出的"AI 软件工程师"，目标是在独立沙箱中完成完整的软件开发任务，几乎不需要人类介入。自主性最高，但也最难预测，适合边界清晰的任务。

> **没有哪条路线是"正确答案"。autonomy 越高，效率潜力越大，但出错时代价也更高。Claude Code 的选择，是在"有实际生产力"和"不失控"之间找到当前技术水平下最合理的平衡点。**
