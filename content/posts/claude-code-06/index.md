# Agent 在何时、以何种方式把控制权交还给你？

全自动执行与随时打断之间，Claude Code 设计了三类精准的人类介入点——让用户始终掌握真正重要的决策，同时不为鸡毛蒜皮的小事打扰你。

## 6.1 三类交互点：何时暂停，何时继续

Human-in-the-Loop（HITL）的核心设计挑战是：交互太多，Agent 失去价值；交互太少，用户失去掌控。Claude Code 的解法是把介入点精确收窄到三类场景，在这三类之外全部自动推进。

上方的流程图展示了这三类交互点在 Agent 执行流中的位置。

### 第一类：强制确认（不可逆操作前）

凡是执行后无法撤销的操作，Agent 会强制暂停等待确认。这不受任何权限配置影响——哪怕用户已经批量授权了所有写入操作，不可逆的 Bash 命令依然会单独弹出确认。

**文件或目录的永久删除**

`rm -rf`、清空目录等操作。即使是 `dist/` 这类构建产物，也需要确认——因为有些 CI 场景 `dist/` 里有不可再生的内容。

**外部系统的写操作**

调用 API、推送到远端仓库、写入数据库、发送通知。这类操作影响范围超出本地，且往往触发不可逆的远端状态变化。

**环境级别的变更**

修改系统配置、安装全局依赖、变更环境变量。这类操作影响的不只是当前项目，而是整个开发环境。

### 第二类：主动澄清（需求歧义时）

当 Agent 识别到需求有多种合理解释，且不同解释会导致显著不同的结果时，它会主动停下来提问。这不是"做不到"，而是"做哪个"的问题。

> **好的澄清问题有三个特征**：提供默认选项（降低用户决策成本）、解释不同选择的后果（让用户有判断依据）、只问一个问题（不同时抛出多个问题让用户应付）。

### 第三类：进度汇报（阶段完成后）

对于多阶段的长任务，Agent 在每个阶段完成后会主动汇报当前状态，让用户有机会确认方向是否正确，或在继续之前调整策略。这是一种"软"的人类介入——用户可以直接让它继续，也可以借这个机会修正。

> **三类介入点的精髓是"非对称保护"**：自动推进低风险操作（读取、分析、局部文件修改），强制介入高风险操作（不可逆、跨边界、歧义）。这样的设计让 Agent 在绝大多数时间里是流畅的，只在真正重要的时刻停下来。

## 6.2 流式输出与透明度设计

Claude Code 采用流式输出——Agent 的每一步推理和操作都实时显示，而不是等所有工作完成后一次性输出结果。这个设计看似是 UX 细节，实则是 HITL 哲学的重要组成部分。

### 为什么流式输出很重要

| 价值         | 说明                                                         |
| ------------ | ------------------------------------------------------------ |
| 实时知情权   | 用户随时能看到 Agent 在"想什么"和"做什么"，不必等待一个黑盒的最终结果 |
| 早期干预机会 | 如果 Agent 在第二步就走错了方向，流式输出让用户能立刻发现并中止 |
| 信任建立     | 透明的执行过程让用户逐渐理解 Agent 的决策模式，从而更有把握地给予授权 |
| 调试便利     | 当任务出错时，完整的执行轨迹让用户能精确定位问题出在哪一步   |

### 输出层次的设计

流式输出并不是把所有内部状态都倾倒给用户。Claude Code 设计了三个信息层次：

| 模式     | 启用方式    | 显示内容                             | 适用场景        |
| -------- | ----------- | ------------------------------------ | --------------- |
| 默认模式 | 直接运行    | 操作摘要 + diff 预览                 | 日常开发        |
| 详细模式 | `--verbose` | 完整 Thought + 工具参数 + token 消耗 | 调试 Agent 行为 |
| CI 模式  | `--print`   | 只输出最终结果到 stdout              | 流水线自动化    |

## 6.3 中途干预：Escape 与方向调整

流式输出带来了一个实时的"紧急停止"能力：在 Agent 执行过程中任意时刻按下 `Escape`，可以中断当前操作。

### 中断后的三种选择

**完全取消**

放弃当前任务，回到命令行。Agent 已经完成的操作不会自动回滚——需要手动 `git checkout` 或其他方式恢复。这是中断后需要注意的关键点。

**修改后继续**

向 Agent 说明新的约束或方向修正，然后让它从当前状态继续。不需要从头开始——Agent 保留对已完成工作的上下文。

**撤销后重来**

用 `git checkout` 回退已完成的改动，然后用更精确的 prompt 重新启动任务。适合发现方向根本性偏差的情况。

> **注意**：中断不等于回滚。`Escape` 只停止 Agent 的后续行动，不撤销已经执行的文件写入或 Bash 命令。在高风险任务中，建议在开始前先确认当前状态已经 `git commit`，这样无论何时中断都有干净的回退点。

### 方向调整的最佳时机

不必等到 Agent 完全"跑偏"才中断。流式输出让你能在以下几个关键节点做预防性干预：

- 看到第一个工具调用的目标文件不对时
- 看到 Thought 里的推理前提有误时
- 看到 diff 里改动范围超出预期时

提早一步干预，比事后修复省力得多。

## 6.4 Headless / CI 模式：无人值守的自动化

上面讲的所有交互点，都预设了"有人在旁边看"。但 CI/CD 流水线里没有人——代码审查、自动测试修复、文档生成，这些任务需要 Agent 在完全无人值守的环境里独立完成。这就是 Headless 模式的设计动机。

### 核心标志与行为

`--print`（或 `-p`）标志让 Claude Code 进入非交互模式：所有确认提示被跳过（需配合 `--dangerously-skip-permissions`），最终输出写入 stdout，退出码反映任务成功或失败。

### 结构化输出：`--output-format json`

在 CI 里，Agent 的输出不是给人读的，而是给下游工具处理的。`--output-format json` 让 Claude Code 输出一个结构化的 JSON 对象：

| 字段         | 说明                                                       |
| ------------ | ---------------------------------------------------------- |
| `result`     | Agent 的最终输出文本，可直接作为 PR 评论、Slack 通知的正文 |
| `cost`       | 本次任务消耗的 token 数量和费用估算，用于成本监控          |
| `session_id` | 会话唯一标识，方便在日志系统中追踪完整链路                 |

### GitHub Actions 集成示例

```yaml
- name: Claude Code Review
  run: |
    claude -p \
      --output-format json \
      "Review the diff in $PR_DIFF, list any security issues" \
      | jq '.result'
```

### 典型 CI 场景

| 场景             | 描述                                                         |
| ---------------- | ------------------------------------------------------------ |
| PR 自动代码审查  | 每次 PR 触发时运行 Claude Code 分析 diff，将审查意见直接发布为 PR 评论 |
| 自动修复失败测试 | 主分支测试失败时自动触发，Claude Code 分析错误、修复代码、提交 fix PR |
| 文档自动同步     | 代码变更合并后，自动更新 API 文档、CHANGELOG、README 中的相关章节 |
| 安全扫描         | 定时扫描代码库，识别潜在安全漏洞、硬编码凭证或不符合安全规范的代码模式 |

> **Headless 模式是 Claude Code "Terminal-first" 哲学的自然延伸**——命令行工具天然适合被脚本化和自动化。这让 Claude Code 不只是一个开发辅助工具，而是可以成为整个工程流水线里的一个可编程节点。

## 小结

模块六的核心是"精准介入"：

- **三类交互点** 解决的是"何时停"
- **流式输出** 解决的是"让用户看到发生了什么"
- **Escape 机制** 解决的是"何时可以打断"
- **Headless 模式** 解决的是"何时完全不需要人"

四者共同构成了一套完整的人机协作框架——在自动化效率与人类掌控之间，找到了对不同场景各自最优的平衡点。
