用户任务输入
用户给出了一个目标，没有任何其他说明。Agent 要做的第一件事，不是立刻动手，而是理解任务的边界。
用户帮我给这个项目加上 GitHub Actions CI，要求：push 时跑测试，main 分支合并时额外跑 lint 和构建。
Thought先读 CLAUDE.md 和 package.json，确认这个项目用什么测试框架、什么 lint 工具，有没有已有的 CI 配置。
ActionRead CLAUDE.md，Read package.json，LS .github/
Observation测试：vitest，lint：eslint，构建：tsc && vite build，无现有 CI 配置。
关键习惯：Agent 拿到任务后的第一个动作永远是"读"，而不是"写"。摸清环境，才能制定合理计划。
复杂度判断
Agent 内部做了一次隐式的复杂度评估。这不是一个明确的"步骤"，而是 Thought 阶段推理的一部分。
Thought这个任务涉及：① 创建 workflow 目录结构；② 写 push 触发的测试 workflow；③ 写 main 分支专用的 lint+build workflow；④ 验证 YAML 语法；⑤ 检查 Node 版本。需要多步骤，属于中等复杂度任务，可以分解后并行处理。
ActionBash: node --version（确认本地 Node 版本，CI 中要对齐）
ObservationNode v20.11.0 → CI 中使用 node-version: '20'
Agent 不会对"写一行注释"这种简单任务做分解——直接执行更快。只有当任务涉及多个独立组件、或需要多轮工具调用时，才值得花推理资源做规划。
任务分解
Agent 把大任务拆成了三条独立的执行线索，每条线索可以作为子任务并行展开。
Thought可以并行处理：
• 子任务 A：创建目录 + 写 push-test.yml
• 子任务 B：写 main-ci.yml（lint + build）
• 子任务 C：读现有测试确认命令是否需要参数
三者无相互依赖，交给 Task 工具并发执行。
任务分解的本质：识别出"哪些步骤可以同时进行"。这跟写代码时识别并发机会是同一种思维。
并行执行（Task 工具）
Agent 调用三个 Task 工具，将子任务委托给独立的 Agent 实例，主 Agent 等待所有结果返回后汇总。
ActionTask A：创建目录 + 写 push-test.yml（含 node setup、cache、vitest --run）
ActionTask B：写 main-ci.yml（on: push: branches: [main]，运行 eslint + tsc + vite build）
ActionTask C：读 vitest.config.ts，确认配置中是否有 reporter 或特殊参数
Observation三个 Task 均完成。Task C 发现 vitest.config.ts 里有 reporter: 'verbose'，需要加 --reporter=verbose 参数。
并行执行的收益：三个 Task 同时进行，总耗时约等于最慢的那个，而不是三者之和。
遭遇歧义 → 询问用户
执行过程中 Agent 发现了一个无法自行决定的问题：用户说"push 时跑测试"，但没说是否包括 PR。
Thought"push 时跑测试"有三种理解：① 任何分支 push；② 仅主要分支；③ 包含 PR。三种语义对应不同触发器，后果不同。无法自行猜测，需澄清。
询问用户"push 时跑测试"——是所有分支 push 都触发，还是只限 main/dev，或者包含 PR？默认给你配"所有分支 push + PR"，但如果你想限制分支告诉我。
用户回复所有分支都跑，PR 也要触发。
Action更新 push-test.yml：on: [push, pull_request]，不限制 branches。
宁可停下来问一句，也不自行猜测然后产生"不太对"的结果。
错误恢复
写完配置后，Agent 尝试用 actionlint 验证 YAML 语法，结果报错。
ActionBash: npx actionlint .github/workflows/*.yml
Observation错误：main-ci.yml:12: "runs-on" is required——漏写了 runs-on 字段。
错误恢复Thought：这是语法错误，直接修复，不需要询问用户。
ActionEdit main-ci.yml：补上 runs-on: ubuntu-latest
ActionBash: npx actionlint .github/workflows/*.yml（再次验证）
完成两个 workflow 文件均通过 actionlint 验证，向用户汇报完成。
语法错误、typo → 自行修复；影响用户资源或权限的决策 → 询问用户。
