高 token 消耗通常是信号，而不是正常现象。触发因素：① CLAUDE.md 缺失，Agent 需要自己探索项目结构；② prompt 不够精确，Agent 需要读更多文件来确认边界；③ Agent 陷入了错误方向的循环。
JSON 输出与 CI 监控：程序化捕获执行数据
--output-format json 让每次 Claude Code 执行都输出结构化数据，可以被监控系统捕获和分析。
# GitHub Actions：捕获执行数据并写入监控
- name: Run Claude Code + Capture Metrics
  run: |
    RESULT=$(claude -p \
      --output-format json \
      --dangerously-skip-permissions \
      "$TASK_PROMPT")

    # 提取关键指标
    INPUT_TOKENS=$(echo $RESULT | jq '.cost.input_tokens')
    TURNS=$(echo $RESULT | jq '.num_turns')
    DURATION=$(echo $RESULT | jq '.duration_ms')

    # 写入监控（示例：InfluxDB）
    curl -X POST "$INFLUX_URL/write" \
      --data "claude_code,task=$TASK_TYPE \
        input_tokens=$INPUT_TOKENS,turns=$TURNS,duration=$DURATION"

    # 输出给下游步骤
    echo $RESULT | jq '.result'
// JSON 输出结构
{
  "result":    "Fixed 2 failing tests in src/auth/token.ts",
  "cost": {
    "input_tokens":       12847,
    "output_tokens":       1203,
    "cache_read_tokens":   8420
  },
  "duration_ms":  34521,
  "session_id":   "sess_01XYZ",
  "num_turns":    8
}
把 num_turns 和 input_tokens 的长期趋势写入 Grafana，当某类任务的 turns 突然从 8 升到 25，通常意味着 CLAUDE.md 过时了，或者代码库结构发生了 Agent 没有感知到的变化。
Golden Dataset：系统化回归测试
精心设计一批"黄金测试任务"，包含明确的验收标准。每次配置变更后跑一遍，对比新旧结果，防止退化。
golden_dataset/
├── task_001_fix_null_pointer/
│   ├── setup.sh          # 准备初始代码状态
│   ├── prompt.txt        # 任务描述
│   ├── validate.sh       # 验收脚本
│   └── baseline.json     # 历史基准（turns、tokens）
├── task_002_add_endpoint/
└── task_003_refactor/
#!/bin/bash
# validate.sh — 语义验证，不比较文本

# 1. 功能正确性
npm test -- --reporter=json | jq '.numFailedTests == 0'

# 2. 无新增 lint 错误
eslint src/ --format json | jq '.[].errorCount == 0'

# 3. 改动范围符合预期
git diff --name-only | grep -v "^src/auth/" | wc -l | \
  xargs -I{} test {} -eq 0

# 4. 效率指标不超出阈值
TURNS=$(cat result.json | jq '.num_turns')
test $TURNS -le 15  # 超过 15 轮视为效率退化

echo "All validations passed"
Golden dataset 的价值在于"可重复对比"。单次跑通不代表什么；但如果上次 12 轮完成的任务这次需要 28 轮，而且 tokens 增加了 3 倍，说明某些地方发生了退化——可能是 CLAUDE.md 被改坏了，也可能是代码库结构改变导致 Agent 需要更多探索。
